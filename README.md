# MarketScan GLP-1 and DPP-4 to GLP-1 Switcher Build

This repository is a restricted-data-safe code and documentation handoff for MarketScan GLP-1 work.

## AI-Assisted Development Boundary

AI-assisted work in this repository is limited to code, documentation, public
code lists, configuration templates, schemas/field names supplied by the
analyst, and synthetic tests. The analyst runs all jobs that touch restricted
MarketScan locations, including:

- raw data under `/N/project/Marketscan1`;
- restricted derived data under `/N/project/mscan_trial/trial_users/tgian`;
- metadata extraction jobs against restricted files;
- production Slurm jobs for Stages 01 and later.

Production outputs described in this README are generated by analyst-run jobs
inside the approved HPC workspace. The AI assistant should not run commands
that read, scan, summarize, or inspect restricted data paths unless the analyst
explicitly requests that action for a specific command.

Development validation by the AI assistant uses only synthetic data and public
metadata/code-list artifacts. A prior footer/schema check of Stage 01 output
files read parquet metadata only and did not load row-level records or column
values; going forward, even metadata/footer checks on restricted paths should
be run by the analyst, with only safe aggregate/log information shared back.

It currently contains:

- `01_build_glp1_pool.R`: a narrow DuckDB/R script that builds a 2022 GLP-1 initiator pool from MarketScan pharmacy and enrollment parquet shards.
- `inputs/glp1_ndc11.csv`: an external GLP-1 NDC11 code list used by `01_build_glp1_pool.R`.
- `code_lists/`: public, source-backed drug NDC and ICD-10-CM concept files generated for the DPP-4 to GLP-1 pipeline.
- `tools/collect_marketscan_metadata.py`: a metadata-only schema collector for local MarketScan files.
- `tools/run_marketscan_metadata_all_years.sh`: a wrapper for collecting metadata from the local all-years parquet layout.
- `slurm/run_marketscan_metadata.sbatch`: a lightweight Slurm wrapper for running the metadata collector on a compute node.
- `R/`: R modules for config validation, synthetic data, drug episodes, enrollment, person-month assembly, and DuckDB I/O helpers.
- `scripts/`: runnable validation, synthetic-data, synthetic-pipeline, Stage 01 drug-fill extraction, and test commands.
- `tests/`: synthetic and metadata-only tests.
- `codex_pack_dpp4_glp1/`: a metadata-only operational pack for building a DPP-4 to GLP-1 switcher person-month dataset on IU Quartz/RED.
- `AGENT_HPC_HANDOFF.md`: read-this-first instructions for a new Codex or AI coding agent session on HPC.
- `MARKETSCAN_METADATA_REQUIREMENTS.md`: the metadata checklist needed before running the full exercise against MarketScan.

The synthetic run exercises:

- exact column mapping in a local config file;
- whether continuous medical enrollment is represented by enrollment spans alone or a specific flag;
- active pharmacy coverage rule, expected to use `RX = 1`;
- exact patient out-of-pocket formula, likely `COPAY + COINS + DEDUCT`;
- whether to use `PAY` or `NETPAY` for each spending endpoint;
- ED, office-visit, outpatient, and inpatient utilization definitions using `STDPLAC`, `BILLTYP`, revenue codes if present, and procedure fields;
- final clinical review of ICD-10-CM diagnosis concept lists;
- whether tirzepatide is included in the primary GLP-1-like exposure or handled separately;
- QC suppression rules for any aggregate outputs.

## DuckDB Direction

Production processing should use DuckDB over Parquet through R. The repository now has `R/duckdb_io.R` with connection and Parquet helper functions. The intended production pattern is staged:

1. scan only necessary pharmacy columns and join to NDC lists;
2. write reduced diabetes-drug fill intermediates;
3. identify candidate DPP-4 to GLP-1 switchers;
4. pull enrollment and medical claims only for candidate `ENROLID`s;
5. aggregate to person-month;
6. write final Parquet outputs and QC reports.

This avoids repeatedly scanning the full raw medical files.

## Stage 01: Diabetes Drug Fills

The first production stage is implemented in:

- `R/stage_01_drug_fills.R`
- `scripts/stage_01_extract_drug_fills.R`
- `slurm/run_stage_01_drug_fills.sbatch`
- `tests/test_stage_01_drug_fills.R`

Stage 01 reads the configured pharmacy `D` parquet files, normalizes `NDCNUM`
to NDC11, joins to the public diabetes-drug NDC concept files, and writes a
reduced restricted parquet with one row per matching drug-class claim. It keeps
core fill fields plus pharmacy spending fields needed downstream:

- enrollee ID, fill date, claim year, NDC11, drug class;
- days supply, quantity, metric quantity;
- `PAY`, `NETPAY`, `COPAY`, `COINS`, `DEDUCT`;
- derived patient OOP as `COPAY + COINS + DEDUCT`;
- selected plan/demographic fields when present.

## Critical Data-Use Boundary

Do not commit, paste, summarize, or send row-level MarketScan data to an AI agent or external service.

Safe inputs for AI-assisted development are:

- metadata-only configuration files;
- table and column names;
- schema summaries;
- file-layout patterns;
- code-list file names and provenance;
- synthetic test data;
- non-sensitive aggregate QC counts when permitted by the data-use agreement.

Even for metadata extraction, the preferred workflow is for the analyst to run
the provided scripts and share only the resulting safe summaries or logs.

Unsafe inputs include:

- patient-level claim rows;
- enrollee IDs or hashes derived from real data;
- claim-line dates and cost values from real data;
- rare-cell or small-cell outputs;
- raw parquet, SAS, CSV, DuckDB, or extracted MarketScan data files.

## Current GLP-1 Pool Script

`01_build_glp1_pool.R` is a single-year prototype. It currently assumes:

- raw 2022 parquet files under `/N/project/mscan_trial/data/parquet_raw/2022`;
- pharmacy `D` shards matching `*_d_*.snappy.parquet`;
- enrollment `T` shards matching `*_t_*.snappy.parquet`;
- pharmacy columns `ENROLID`, `SVCDATE`, `NDCNUM`, `DAYSUPP`;
- enrollment columns `ENROLID`, `DTSTART`, `DTEND`, `RX`;
- `RX = '1'` means pharmacy benefit enrollment;
- integer dates, if present, are days since `1960-01-01`;
- output paths under `/N/project/mscan_trial/trial_users/tgian`.

The script:

1. reads the GLP-1 NDC11 list;
2. discovers pharmacy and enrollment parquet shards;
3. validates required columns;
4. standardizes fill and enrollment dates;
5. builds continuous pharmacy-benefit enrollment spells;
6. identifies each enrollee's first GLP-1 fill;
7. keeps patients with pharmacy coverage from 180 days before through 180 days after index;
8. writes `glp1_pool.parquet` and `glp1_rx_pool.parquet`.

Before running it on another HPC project path, update the hard-coded `data_dir` and `work_dir` or refactor it to use a config file.

## Full DPP-4 to GLP-1 Exercise

The broader target is a person-month analytic dataset for patients who cleanly replace DPP-4 therapy with GLP-1 therapy.

Primary design:

- data years: 2017 through 2023;
- index window: January 2018 through December 2022;
- baseline: 12 months pre-index;
- follow-up: 12 months post-index;
- event months: `-12` through `+12`;
- exposure: first qualifying GLP-1 fill after 365-day GLP-1 washout;
- required source therapy: DPP-4 use before index;
- primary cohort: `clean_replacement`;
- output grain: one row per enrollee episode per event month.

The detailed operational design is in:

- `codex_pack_dpp4_glp1/AI_Agent_Operational_Brief_DPP4_to_GLP1_MarketScan_Quartz.md`
- `codex_pack_dpp4_glp1/config_template_dpp4_to_glp1_marketscan.yaml`
- `MARKETSCAN_METADATA_REQUIREMENTS.md`
